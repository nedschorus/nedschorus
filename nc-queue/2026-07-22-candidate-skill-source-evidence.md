# Candidate-skill source evidence — walk input for outer items 2–10

Produced during the boss's AFK window (2026-07-22 evening) by six parallel research agents commissioned by new-vp; every source named in pair #9's shortlist was read in full. Evidence caveats are stated per candidate: most sources were fetched raw/curl-verbatim; the two anthropic.com engineering articles were readable only through a fetch summarizer (quotes from them are high-confidence but not curl-confirmed). Raw agent reports live in the commissioning session only; this note is the durable extract and the walk input. Recommendations are new-vp positions, un-ruled.

Recurring cross-candidate findings, worth one walk minute up front:

- **Two gaps recur in nearly every source and must be authored by NC, not borrowed:** an explicit safe-permissions statement (what the skill may read/write), and a checkable output schema. No official source provides either consistently.
- **The Superpowers pattern:** both files carry a high-quality mechanical kernel wrapped in absolutist mandates (Iron Laws, delete-and-start-over). Adopt the kernel, strip every ALWAYS/NEVER/delete instruction — matching the ruled explain-why register.
- **Two candidates get near-wholesale official spines** (design-change ← Anthropic code-architect; review-change ← OpenAI review-agent); the rest are compositions.

## Item 2 — `define-work`

Sources: OpenAI `define-goal` SKILL.md (A, raw-verbatim); Anthropic "Let Claude interview you" best-practices section (A, guidance-not-contract). Adoptable spine from define-goal: the self-gating trigger ("If the user only asks for ordinary implementation work, do the work directly"); the six-field restatement (outcome, artifact/system, verification method, in-scope, out-of-scope, stop-condition); the five-question Goal Quality Bar as the skill's own checkable exit test; repair-or-ask-once discipline; rejection of pure-activity framings. From the interview tip: the negative trigger ("If you could describe the diff in one sentence, skip the plan"), the spec self-containedness bar (files, out-of-scope, end-to-end verification step), and the produce-then-fresh-context-executes handoff. Reject: goal-tool state coupling (get_goal/create_goal), the six-domain quantification prose, unbounded "interview until we've covered everything" (fuse with ask-once into a bounded question loop — this fusion exists in neither source and is the skill's main authored piece). Gaps to author: assumptions as a first-class output (in the shortlist, in neither source); permissions; written-spec format. Recommendation: BUILD from define-goal's contract + the three interview grafts.

## Item 3 — `plan-rewrite-slice`

Source: OpenAI code-modernization cookbook (A for what-the-vendor-prescribes; page fetched and probed). **Decision-critical finding: the source has no defect-handling vocabulary at all** — probes for legacy-bug/incorrect-behavior guidance returned nothing, and its operative default is verbatim "propose the smallest change to the modern implementation that will align it with the COBOL behavior": every parity diff is resolved by making new code reproduce old behavior. The shortlist's caution is understated — the source actively entrenches old defects. Adoptable: bounded-pilot-slice selection; the four separated concerns as required plan sections (current behavior / target design / external contract / validation-parity plan); definition-of-done in the plan; parity harness as evidence; per-behavior traceability to the old source symbol. Reject: parity as the cut-over gate (structurally forbids INTENTIONAL CHANGE, blesses OLD BUG), the align-to-old fix loop as default, phases 0/5 (program scaffolding), mandated filenames, OpenAPI as required artifact. The skill's core authored piece is the inversion: a parity diff triggers the four-way classifier (MUST PRESERVE → align; INTENTIONAL CHANGE → document + update oracle; OLD BUG → do not reproduce, record corrected behavior; UNRESOLVED → park with owner), and the gate is "every diff classified," never "zero diffs." Also to author: oracle-update discipline, evidence standard per classification, concrete slice-boundary tests ("realistic but bounded" is a vibe). Recommendation: BUILD; treat the source as structure donor and named anti-pattern.

## Item 4 — `design-change`

Sources: Anthropic `code-architect` agent (A, double-fetched for quote fidelity); OpenAI ExecPlans (B). code-architect is the backbone, nearly one-to-one with the candidate: read-only toolset, three phases (pattern analysis → commit to one architecture → implementation blueprint), the one-recommendation rule ("Make confident architectural choices rather than presenting multiple options"), file:line evidence grounding, and seven output sections covering every candidate beat (boundaries, interfaces, data flow, failure behavior, likely files, order, validation). Graft from ExecPlans, minimally: behavioral validation-and-acceptance (commands + expected outputs a human can verify) and full repository-relative path discipline. Reject from ExecPlans: novice self-containment, the thirteen-section living-document skeleton, execution-tracking ceremony, unbounded length — exactly the flagged over-weight. Gaps to author: explicit read-only guardrail in prose, "assume a competent reader of this repository" line, a one-line alternatives-considered rule, length ceiling. Recommendation: ADOPT code-architect's contract as the spine with the two ExecPlans grafts.

## Item 5 — `write-test-plan`

Source: Anthropic `pr-test-analyzer` (curl-verbatim). Agent's grade: **B for this purpose, disagreeing with the shortlist's implicit higher rating** — the rubric is retrospective (analyzes an existing diff + existing tests) and phase-mismatched with plan design. What inverts cleanly to before-the-fact: the consequence-ranked 1-10 criticality scale (which requirement/risk gets a test first); "provide specific examples of failures it would catch" (→ the observable-oracle + expected-failure trace columns); the gap taxonomy (error paths, boundaries, negative validation, concurrency) as up-front case classes; behavior-over-implementation as an oracle constraint; trivial-code exclusion (→ justified-exclusion leaf). What does not invert: the whole read-the-diff analysis process, brittleness review of unwritten tests, positive observations. Coverage vs the target trace (requirement → level → setup → stimulus → oracle → expected failure → command → automated/human → exclusion): the source supplies roughly the right half; **test level, setup/stimulus separation, exact command, automated-vs-human, and requirement→test traceability must all be authored** — Waza's `spec verify` requirement-ID mapping (item 10 below) supplies the traceability method. Recommendation: BUILD (the shortlist's "no adoptable source" call is confirmed); import the scale + oracle framing.

## Item 6 — `attack-artifact`

Sources: Anthropic adversarial-review best-practices section (A); Osmani `doubt-driven-development` (B, second-fetch verbatim). **The isolation invariant is confirmed independently by both:** the reviewer sees artifact + criteria only, never the author's reasoning (Anthropic: "sees only the diff and the criteria you give it, not the reasoning that produced the change"; DDD as hard input contract: "Never pass CLAIM, reasoning, or conclusions"). From Anthropic also: the three-part prompt (name the work, the plan to check against, what counts as a finding) and the calibration guard ("A reviewer prompted to find gaps will usually report some, even when the work is sound"). From DDD: the flaw taxonomy (unstated assumptions, edge cases, hidden coupling, contract violations, failure modes under unexpected input); the clean-result mandate ("Do NOT validate. Find issues, or state explicitly that you cannot find any"); the four-way reconcile precedence (contract-misread / valid-actionable / valid-tradeoff / noise); bounded stop conditions. Reject: DDD's per-cycle cross-model offer and CLI plumbing, the magic 3-cycle constant, code-centric triggers. Gaps to author: the intent-gaming check (criteria satisfiable while violating intent — the candidate's sharpest requirement, present in NO source), untestable-claim detection, output schema. Recommendation: BUILD as the fusion; author the intent-gaming check explicitly.

## Item 7 — `implement-with-evidence`

Source: Superpowers `test-driven-development` (curl-verbatim; A for the evidence kernel only). Adopt the red/green evidence contract exactly: watch the new check fail first ("If you didn't watch the test fail, you don't know if it tests the right thing"); confirm it fails for the intended reason (feature missing — "not errors," not typos); smallest code to green; verify green + pre-existing checks + pristine output; the three failure-triage rules (immediate pass → fix the test; error → not yet valid evidence; green breaks others → fix code, not test). Reject, enumerated verbatim in the full report: every delete-and-start-over mandate ("Write code before the test? Delete it. Start over.", "Delete means delete", "Implement fresh from tests. Period.", the Red-Flags delete list, "Can't check all boxes? You skipped TDD. Start over."), the Iron Law framing, and the rationalization-table absolutism — the shortlist's caution confirmed line-by-line. Gaps to author: generalize "test" to any observable failing check (build, assertion script, repro); define the *relevant* regression subset instead of absolute all-tests; a judgment-based exception path. Recommendation: BUILD small, kernel-only.

## Item 8 — `diagnose-failure`

Sources: Anthropic best-practices symptom/verification rows (B — principles, not procedure); Superpowers `systematic-debugging` (curl-verbatim; A). **The shortlist's open concern is answered with a quote: the finite stopping condition exists** — "If ≥ 3: STOP and question the architecture… DON'T attempt Fix #4 without architectural discussion"; the loop is bounded at three fix attempts and terminates into escalation, with the counter on fixes, not hypotheses. Adopt systematic-debugging's spine: read errors fully → reproduce consistently ("If not reproducible → gather more data, don't guess") → per-component-boundary instrumentation (the standout concrete technique) → single hypothesis ("I think X is the root cause because Y") → one variable minimally → single fix at source with failing test first → verify. Adopt from Anthropic the front end: paste the exact symptom text, name the likely location, show evidence rather than asserting success. Reject: the ALWAYS/Iron-Law absolutism (rigidity to soften, not hazards — nothing destructive found here). To author: promote the buried "No Root Cause" clause into an explicit, labeled emergency-mitigation-vs-root-cause distinction (the candidate requires it first-class); remap "discuss with your human partner" to NC's escalation target. Recommendation: BUILD from the composition; this pairs with the quarry's hard-problem method at walk time.

## Item 9 — `review-change`

Sources: OpenAI `review-agent` sample (curl-verbatim; A); Google "What to look for in a code review" (A as criteria, not workflow). **review-agent satisfies every candidate requirement nearly wholesale:** read-only defect-first stance; exact-revision anchoring (merge-base + diff); the five-part finding gate (meaningful impact / discrete+actionable / introduced by this change / scenario demonstrable from code / author would fix it); `[P0–P3] imperative title — file:line` + one-paragraph scenario output; and the clean-result rule verbatim: "If there are no qualifying findings, say `No findings.` Do not invent a finding to fill the result." Google layers under it as a criteria checklist (Design > Functionality > Complexity > Tests > Naming…, "Look at every line," broad context, Good Things as positive-output license) — reject its use as workflow (it has no procedure or severity model) and subordinate style checks. Checked detail: the "not perfect, just better" principle is NOT on the looking-for page (lives elsewhere in eng-practices) — do not misattribute. Gaps: severity calibration beyond four labels; the no-test-suite case. Recommendation: ADOPT review-agent as the spine, Google as subordinate criteria. Note for the walk: NC already has d-review for documents; this candidate is its code sibling — the shared five-part finding gate and clean-result rule suggest a common finding-contract across both.

## Item 10 — `eval-agent-change`

Sources: Anthropic evals article (A, summarizer-mediated); Anthropic April-23 postmortem (B as skill source, A as motivating evidence; summarizer-mediated); Microsoft Waza README (A, curl-verbatim). From the evals article: capability-vs-regression split (regression holds ~100%); balanced positive AND negative trigger cases; harness isolation on real builds; grade the output, not the path; pass^k concept for must-be-consistent behaviors; evals built before the change lands. From the postmortem, the non-negotiables (requirements, not steps): per-model evaluation of every runtime the change ships to; exact-production-build testing (interaction effects masked a bug past "multiple human and automated code reviews… unit tests, end-to-end tests… and dogfooding"); line-by-line ablation of prompt changes. From Waza, the smallest useful adoption — a method and a file convention, never the platform: (1) the baseline-vs-candidate A/B method (each task twice, delta computed — reimplementable in ~20 lines against our runtime); (2) the positive-trigger-*/negative-trigger-* task file convention; (3) the `spec verify` discipline (parse a skill's USE FOR / DO NOT USE FOR into requirement IDs, map each to a task — also feeds item 5's traceability gap). Reject: the Go/Copilot runtime, LFS binaries, Azure machinery, org-scale eval teams. Recommendation: BUILD lean around those three Waza pieces + the split; this candidate connects directly to the skill-creation deep-dive's test doctrine (§3) and to the deferred walk-me scenario tests — one eval substrate can serve all three.
