# nedschorus Boot-Up Plan — the boss's multipoint plan, annotated

**What this is:** the plan for booting nedschorus — the preparation work the boss + new-vp do to bring choirmaster into existence. Structure and order are the boss's (proposed in the predecessor session, restated here); annotations note what each step produces. The first task choirmaster will run is deliberately the LAST step — it is almost irrelevant to the boot-up itself. For the boss's markup. Quarry-side planning doc; only files that ship into nedschorus are written for a zero-context reader (a reader who has only CLAUDE.md and the file itself — no session history, no rationale).

## Standing decisions the plan assumes (boss-ruled 2026-07-20)

- Fresh sibling repo `~/Projects/nedschorus`; the old project is the QUARRY, read-only; every import crosses the entry checkpoint (`entry-manifest.md`: quarry SHA + one-line purpose + date, same commit).
- Primary agent: **choirmaster** (Claude runtime). A **Codex companion** joins it — execution mass, cross-runtime review — talking over **mini-comms** (a second instance of the two-log-file bridge pattern). Companion timing not yet ruled.
- 100% manual edit-production pipeline; automation re-admitted step by step (manual → script-you-run → automation, each earned).
- Bridge and mini-comms logs `.gitignored`; durable content promotes to committed files via handoff.
- Choirmaster's first task will be something from the quarry's `git-clean-slate-plan.md`.

## The steps

**1. Assemble the starting skills.** The set choirmaster boots with FIVE skills:

- **d-review** and **walk-me-through** — trims of the quarry versions (fleet doctrine references — postal, trailers, worktree plumbing — removed or inlined), through the entry checkpoint.
- **The handoff skill** (`fast-handoff` now, renamed `handoff` at entry; full design: [fast-handoff-design.md](fast-handoff-design.md)). The ceremony is a SKILL; the kernel carries only the one-line successor-side pickup (run the read command; follow only on a fresh read).
- **`ghi-write`** (issue-only artifacts: bug reports, PR descriptions, discussions) and **`md-write`** (the MD-GHI pair: every durable markdown is born with its issue — see Project organization; triggers include the boss's word "memorialize", carried in the frontmatter description) — new, from the boss's ChatGPT design session (2026-07-21). Each is SMALL and directly embeds the zero-context-reader rule (write for a competent reader with repository access who has NOT seen this conversation) plus a composed-in check before returning — generate → reader-context check → minimally revise — never a second skill the user must invoke (indirection rejected in that session). The check is three tests: could a stranger identify the subject; could they understand why this exists (one sentence usually suffices); could they act without the conversation. Names are chosen for skill-selection matching — the name and frontmatter description are the triggering interface, the body is the behavior.

Product: the skill files, boss-read. Note for step 8: the Codex companion gets `ghi-write`/`md-write` as Codex-side skills too (per the same session: AGENTS.md notes cannot reliably override the system prompt's concision push — skills are the working control point on both runtimes).

**2. Revise CLAUDE.md for them.** The kernel, written under the naked-reader calibration: per sentence — training covers it → cut; training silent (project facts) → state plainly; training conflicts → NOT/DO override. No rationale, no history, present-tense truth only. The existing seed draft is pre-calibration input, not the base. Product: the kernel, line-by-line boss-admitted. **The zero-context-reader writing rule for every agent-read durable artifact (boss insight, 2026-07-21): write it as if it were a subagent prompt** — the artifact must let a fresh subagent act correctly with no other context, and the test is literally handing it to one (a fresh subagent's context floor — CLAUDE.md + prompt — is the measured boot state of any future reader). Ruling (boss, 2026-07-21): **agent-readability outranks boss-readability in durable artifacts** — the system is built for autonomy, the boss reads through agents on demand ("explain this to me"), the same way he treats code; artifacts are optimized for the agent context that consumes them. Delivery of the rule, so it fires at writing time: one kernel line, plus the same instruction embedded in every skill or prompt that produces a durable artifact (MD file, GHI, handoff): written as a subagent prompt for a zero-context reader + pinned SHAs and dates + no live status that rots. The founding-phase exception is workflow, not doctrine: documents the boss line-by-line admits (this plan, the kernel drafts) are written for his direct read.

**3. Revise the wiki for them.** The boss + new-vp walk the pages together and eliminate roughly half; survivors are cleaned for the naked reader (false-in-nedschorus mechanics removed, teaching kept) and imported through the checkpoint. This is our job, not the naked agent's — utility judgment needs context they will not have. Product: the starting wiki set, in the Obsidian shell.

**4. Build the minimal comms bridge — and make it just work, tested.** The two-log-file bridge between choirmaster and the old system (new-vp), per `comms-bridge-spec.md`; the same pattern stamped a second time as mini-comms for the choirmaster↔companion pair. This step creates the bare repo skeleton as its test bed (`git init`, `.gitignore` with the log paths, empty `entry-manifest.md`); full populate waits for step 6. Tested means tested before boot: a genuinely naked subagent, handed only the draft kernel, must actually use the bridge — trained-never-to-ask is the known risk, and the countermeasure is mechanical triggers (write intent to the bridge before any quarry import or new design; append open questions at end of task). Possible free rider, unruled: each side arms a harness-native Monitor on the other's log for idle-wake — no daemons.

**5. Write their one-time startup instruction.** What a fresh choirmaster session reads first, exactly once: what to read in what order (kernel → HANDOFF.md → quarry guide), nothing else. Written for the zero-context reader throughout — a list of files to read, no why.

**6. Set up their worktree and git, and populate.** The git substrate (identity, `useConfigOnly`, base config — done by us, minutes); `.claude/settings.json` with `{"permissions": {"defaultMode": "bypassPermissions"}}` — choirmaster runs with bypass permissions on (boss ruling 2026-07-21; the JSON setting is durable and beats a launcher flag; the old fleet instead runs allowlists, which still leak prompts). Bypass means no permission prompts at all — the guards are git reversibility and the boss reading everything; nothing mechanical then stops a quarry write except the kernel rule, and whether a settings `deny` rule for the quarry path still binds under bypass mode is UNVERIFIED — probe at populate if wanted. Plus: the Obsidian wiki shell holding step 3's survivors, the skills from step 1, the kernel from step 2, the boss launcher scripts (manifest row 9: fresh start / `claude --continue`, focus-restore cribbed from the quarry), and the founding handoff — written LAST, at cutover, so it captures true state.

**7. Pick ONE task for them to do.** One, because we learn from it how to help them do the second. Ruled direction: something from `git-clean-slate-plan.md` — they are trained on git but have never been given a fully set-up git with rules, both prompts and code. Exact slice picked here, at the end, when the boot-up has taught us what they can absorb. Boot, run it, learn; then task 2.

## Project organization — the three committed homes (boss direction 2026-07-21 + new-vp amendments, PROPOSED until marked up)

The old system's `docs/working/` junk drawer — no lifecycle, things rot in place, "scratch, not doctrine, do not cite" — is the failure this rule prevents. nedschorus commits markdown to exactly three homes:

1. **`docs/wiki/`** — permanent truth (the curated Obsidian shell from step 3). Graduation target for everything that outlives its work.
2. **`docs/issues/<number>-<slug>.md`** — working material, **one document per substantial GitHub issue**, the number making the tie unambiguous and grep-able from either side. Path stays stable for the issue's life (state never lives in the path — no draft/accepted/done directory moves; renames are the hardest merge case). The file's header carries the issue link ONLY — **no hand-stamped `Status:` line**: the issue's own open/closed state is the single status authority, and a hand-maintained copy of it in the file is the drift surface we have measured repeatedly (amendment 1). At issue close, the SAME commit disposes the document — delete it (working material), or promote the lasting part to `docs/wiki/` — tying disposal to a mechanical moment instead of a someday judgment (amendment 2); the GHI itself carries the outcome. Stragglers surface via the boss-review-log.
3. **`handoff/`** — the numbered handoff series (plus `boss-review-log.md` at the repo root, the boss's file).

Everything else is uncommitted scratch (session scratchpads, bridge logs — already gitignored). No `docs/working/` is ever created.

**The fourth home — `docs/cross-project/` (boss-ruled 2026-07-21; directory name new-vp's pick, rename freely):** pre-planning and shared nedschorus artifacts live IN nedschorus, in one clearly defined place designed to be shared across the two projects — there is no place for them in nedlern. This directory holds artifacts that both systems read or that the old system authors about the new one: the founding plan (this file), the boot-pack drafts, the comms bridge spec, the fast-handoff design. The repo is live at https://github.com/nedlern/nedschorus; these documents migrated here 2026-07-21 from the quarry's `docs/working/next-system-boot-pack/` (quarry copies frozen with a pointer). Choirmaster reads this directory on demand only — never at boot; the boot surface (kernel, wiki, skills, startup files) stays zero-context-clean, which is what survives of the earlier planning-stays-in-the-quarry rule.

**Session provenance on durable artifacts (boss direction 2026-07-21):** every artifact records which session wrote it, because the session id is now a working pointer — `handoff.py transcript <session-id>` recovers the writing session's full dialog when context was lost. Mechanism split by artifact kind: **MD files** carry it in their commit messages (git-native, per-change, cannot rot in the file; `git log -- <file>` yields the artifact's full session history — the old system's commit-stamp practice, carried forward), never as a frontmatter field that stales on the next edit. **GitHub issues** are not in git, so `ghi-write` injects a body footer line (`Written by session <id>`). This deliberately reverses the old wiki's no-author-stamps rule: that rule was written when a stamp was decoration; the transcript tool makes it load-bearing.

**The MD-GHI pair (boss-ruled 2026-07-21):** the discussion-and-prep state needs a durable artifact before the work concludes, but "should this discussion get an issue?" is a judgment call the boss will forget and agents are poor at. The resolution removes the judgment: **a request for a durable MD IS a request for an MD-GHI pair.** The boss's trigger word is **"memorialize"** — when he says it (or asks for a durable MD), `md-write` produces the pair: the doc at `docs/issues/<n>-<slug>.md` and its issue, born together, never separately. Nobody decides whether a discussion deserves an issue; asking for durability is the signal, and the skill does the rest mechanically. WIP is the pair's normal state — `docs/issues/` is the working home, so revision-in-place is expected, not exceptional.

**The disposition step (inside `md-write`, mechanical-first):** before minting anything, the skill searches existing pairs — glob `docs/issues/`, search open issues — for the subject. Four dispositions: **NEW** (no match → mint pair); **REVISE** (match, same intent → edit the existing doc, comment the issue; the default on an exact-subject match); **REPLACE** (match but this supersedes → new pair, close the old issue with a pointer in the same commit); **REMOVE** (the discussion concluded the thing should not exist → close the issue with the outcome, delete the doc). The skill proposes its disposition with evidence (the matched pair, last-touched date, one-line summary) and proceeds — default to autonomy; a genuinely ambiguous match is flagged to the boss-review-log, never blocked on. The handoff ceremony stays the backstop for un-memorialized discussions: the naked writer reads the transcript, so an undocumented discussion is exactly what its draft surfaces.

**Promotion of productized designs to the wiki (boss direction 2026-07-21, adopted):** when a design ships, its lasting part promotes to `docs/wiki/` at the issue-close disposal moment (the same commit, per amendment 2). The promoted page carries frontmatter `design-as-of: YYYY-MM-DD` plus one standard line: *"The production system is the SSOT; this page describes the design as of the date above — newer code and open issues may have advanced past it."* This prevents the old system's measured reviewer failure (treating a wiki design page as SSOT against newer reality). A future `design-update` skill — reconcile an aged wiki design page to production reality, refreshing the as-of date — is a ladder candidate, not boot-set.

**Amendment 3, a naming decision the boss should rule:** the old system says "GHI"; GitHub and every trained agent say "issue"; one concept must get one canonical term (naming rules). For a zero-context-reader system "issue" is the free word (training already carries it) and "GHI" costs a kernel definition. Lean: directory `docs/issues/`, prose term "issue" — and the `ghi-write` skill keeps its name because the boss's typed vocabulary ("ghi") is the primary trigger, with "GitHub issue" in its frontmatter description so both trigger words match.

**Subsystem structure (boss direction 2026-07-21, sharpened; RULED 2026-07-21 — approved with the boss's restatement: co-locate when free, subdirectories inside pinned roots, subsystem token in names, hygiene-script rung):** every subsystem — and subsystems of subsystems — gets a matching structural identity, so tests match code and design plans match test plans per subsystem, by construction. The workable form at this scale has two parts. (1) **One subsystem token, greppable across all roots:** the harness pins some locations (skills live in `.claude/skills/`, scripts in `scripts/` for PATH-ability), so a single physical directory per subsystem is impossible — instead the subsystem's name appears verbatim in every artifact across roots (`scripts/handoff.py`, `scripts/test-handoff-*.py`, `.claude/skills/handoff/`, `docs/issues/<n>-handoff-*.md`, `handoff/` for its artifacts), and one grep returns the whole subsystem. (2) **Lazy subdirectories per root:** when a subsystem has two or more files within a root, they move into a subdirectory of that root (`scripts/handoff/`, `tests/handoff/`) — created at the second file, never speculatively, so no empty scaffolding. (3) **Checking vs placement (boss question 2026-07-21):** a hygiene script can mechanically check three things — every file carries a known subsystem token (the token set is derived from directories in use, no hand-maintained registry), 2+ same-token files loose in a root should be a subdirectory, tests carry the test marker. It cannot decide which subsystem a new file belongs to — placement is the writer's judgment at creation time, carried by the kernel line and writing skills — and a NEW token (never seen in the repo) is a judgment moment the script routes to the boss-review-log ("new subsystem, or synonym drift?"), never a block. The script is a script-you-run ladder rung, sibling of the entry-checkpoint check (one repo-hygiene script for both); bounded and testable — a candidate slice for choirmaster's early tasks. Not v1.

## Further additions (new-vp, 2026-07-21 — walked with the boss; per-item status marked)

4. **[RULED 2026-07-21: approved, with the boss's single-extraction refinement; folded into fast-handoff-design v9] Commit the denoised transcript into the handoff series — one JSONL extraction per session, ever.** The transcript fallback and every session-provenance pointer dangle if a session JSONL is lost — and `~/.claude` has NO git backup (old-system measured fact: Time Machine only). The flow: at ceremony start the extraction runs ONCE, saving the dialog as `handoff/<NNNN>-transcript.md`; the naked writer reads that file (never a second extraction); `write` commits handoff + transcript together. Every later reader — a confused successor, long-term analysis — just Reads the committed file's tail; no script, no JSONL. The JSONL-extraction path survives for exactly one case: a session that died without running its ceremony (the ALREADY-READ recovery scenario) — its transcript was never committed, so `handoff.py transcript <session-id>` recovers it on demand. Successor's normal boot still reads only the handoff distillation (ruled); the transcript tail is the on-demand fallback. Cost: bounded text per session. This amends fast-handoff-design if approved.

## Companion timing — early, in two stages (boss direction 2026-07-21: "the sooner nedschorus is double headed, the better"; staging PROPOSED)

The reconciliation: separate **parallel cognition** (wanted immediately) from **shared write access** (needs the git foundation). Two stages:

- **8a — early admission, right after choirmaster's boot test passes** (one variable at a time: choirmaster boots alone first, the companion follows within days). Needs: the companion's `AGENTS.md` kernel (the zero-context calibration applies twice, once per runtime), mini-comms live (Monitor-armed on the Claude side; a small codex-side watcher, quarry prior art), and **write partitioning at the worktree level (boss's form, 2026-07-21)**: the companion drafts freely in its OWN clone — any paths, no coordination — and never pushes; the single git-writer to main is choirmaster, which promotes the chosen artifact into the canonical checkout and commits it once (the old system's proven peer-drafts-VP-applies pattern, roles renamed). The companion needs no git identity, no branch rules in this stage. Mode: advisory + proposed artifacts + cross-runtime review — the boss's parallel workflow (both sides propose, cross-compare, best-of-both synthesized by choirmaster) works fully in this stage. **Asking both heads the same thing:** manual rung, usable now — type the ask to one side, tell the other to read the tail of the denoised transcript (never the raw JSONL — noisy, and models are trained not to know it; codex's own logs under `~/.codex/sessions/` serve the reverse direction); script rung — mini-comms carries the relay (ask choirmaster, it appends the task to its log, the companion's watcher picks it up).
- **8b — largely DISSOLVED by the single-writer throat (boss insight 2026-07-21, elaboration PROPOSED):** the companion never graduates to direct git-writer — all check-ins flow through one throat to main, permanently. See the throat section below.

**The single-writer throat — nedschorus's permanent check-in architecture (boss insight 2026-07-21: "at the heart of checking in files"; PROPOSED for confirm):**

- **One identity can push to main. Ever.** Enforced by push rights (branch protection's one surviving job), not by a sanctioned-path convention — the old system's `nedlern-pr-merge` proved the throat shape works and its bypassability proved why enforcement must be structural.
- **The throat is as dumb as possible.** The rote part (validate, place, stamp provenance, entry-manifest check, reviewed-check for gated classes, commit, push) is a SCRIPT — zero context cost, background by nature. A **spawn-per-job background subagent** (cheap model, dies when done) handles multi-step promotions with edges (reading a companion proposal, placement, collision with moved main). No persistent committer subagent: it would die with its session and accumulate lifecycle machinery for no judgment gain. Choirmaster handles only the rare oddity.
- **Review is orthogonal and stays upstream:** cross-runtime review happens before promotion; the throat mechanically refuses unreviewed promotions of designated classes. Quality is a gate the throat checks; serialization is what the throat is.
- **Consequences:** task 1's git slice narrows again (config + workflow rules + throat enforcement; no multi-writer referee machinery); the quarry's multi-writer identity designs (three Apps, credential helper) are likely never imported; the old system's PR-prod plan is NOT retrofitted — this is nedschorus DNA only.

**Future candidate, recorded:** the parallel-proposal workbench — a custom UI for running both runtimes side by side, coordinating mid-discussion, and merging proposed artifacts (boss: too big for now, good early nedschorus build). Mini-comms is its minimal precursor.
