# fast-pr-to-prod — the single-writer throat (specification)

How a change reaches main in nedschorus. Main is production until the first long-running process exists (no deploy step exists yet; one is designed when such a process appears). Actors: choirmaster (the single writer), the companion (drafts in its own clone), the boss. The name is the family name for the re-derived minimal pipeline; ordinary changes use no pull request.

## The throat

- **Exactly one identity can push to main: choirmaster's credential.** Enforced LIVE since 2026-07-21 by branch protection on the org-owned repository (`github.com/nedschorus/nedschorus`, organization `nedschorus`): pushes to `main` restricted to the machine credential (`NedLern`/`nedlern`) alone, enforce-admins on, force-push and deletion blocked. Account layout (boss-ruled 2026-07-21, org form, two owners): `NedLern` — also the machine/agent credential — and `NedLerner` are both organization owners, so either account can recover the org if the other is lost; the push restriction admits only `NedLern`, and enforce-admins is on, so owner status grants no exemption from the restriction list — `NedLerner` has settings and emergency power but no push, and a protection change by any owner is a deliberate, visible act, never a standing path. The boss never commits directly: a boss-originated change is drafted with an agent and lands through the throat like any other. Issues cost nothing: the repository is public, so opening and commenting needs no repository permission. Accepted residual: the machine credential is agent-held AND an org owner, so an agent could deliberately edit the protection — the same cooperative class as a raw push, caught by the trailer-absence audit and the boss's reading; the App-as-sole-writer rung retires it. The remote enforces only this lock; the checks live in the landing script below. The companion has no git identity; anything it produces lands under choirmaster's credential with the companion's attribution stamped in the commit message.
- **One landing command for every change.** `scripts/throat.py land` is how ANYTHING reaches main — choirmaster's own work and companion promotions alike. It runs the checks, writes the provenance stamp (producing session id; for companion drafts, whose draft), commits, and pushes. The push monopoly has ZERO exceptions (boss-ruled 2026-07-21, package-review S2): handoff ceremony commits — `handoff.py write` and read stamps — are LOCAL-only, carry the same provenance trailer schema, and ride to origin with the next landing's push; the founding handoff, written before the throat exists, is pushed by choirmaster's first landing. Choirmaster does not raw-push; the kernel instructs landing through the script, and since choirmaster is the only identity that CAN push, a raw push is a deliberate bypass — a cooperative-model residual the boss's reading catches, not a mechanical impossibility.
- **Choirmaster's own change:** edit in the canonical checkout → `throat.py land` at every coherent change. No branch, no pull request, no parking state.
- **The companion's change:** draft freely in its own clone (it never pushes) → append a **promotion request** to its mini-comms log — a unique companion-generated `request-id`, the draft's path list in its clone with a content digest per listed path, its base commit (the main SHA it drafted against), and at most ONE declared quarry import (source SHA + source path + destination path; a second import is a second request). The digests freeze the request at send time: promotion refuses a path whose current bytes differ from its digest, so an edit after send is a new request, never a silent substitution; the request travels as a versioned machine-parseable request block (entry-body format fixed in `comms-bridge-spec.md`), and every request receives a mandatory disposition entry naming its `request-id` — `landed` with the landing commit, or `refused` with the named defect — so a request is never ambiguous between waiting and resolved (boss-ruled 2026-07-21, package-review S15) → choirmaster picks the request up at its next turn and either runs `throat.py land` directly (single-file, clean apply) or spawns a background promotion subagent for multi-step work (reading a proposal, deciding placement, reconciling with moved main). The routing is choirmaster's call; the subagent runs inside choirmaster's session and therefore wields its credential. Best-of-both synthesis (both heads drafted the same artifact) is choirmaster merging the drafts, then landing the result like any of its own changes.

## The checks — inside `throat.py land`, from day one

Checks run per landing, against the exact tree being landed (the composed result after any reconciliation, not the earlier working tree); a refusal refuses the whole landing, structured and named — what failed, why, the next step. No silent failures.

| Check | Behavior |
|---|---|
| Provenance | The script WRITES the stamp itself — injected, not remembered — as a fixed parseable trailer schema: base SHA, producing session and runtime, drafts' attribution, `request-id` (for promotions), task/issue id, declared imports with destination paths, review evidence, and the list of checks that ran (readable with `git interpret-trailers`; the trailers ARE the machine-readable record). Machine-determinable fields are injected; fields the script cannot determine (task/issue id, review evidence) arrive as named `land` arguments or from the promotion request. The schema is an exact grammar (boss-ruled 2026-07-21, package-review S16): every field is either required or explicitly empty — a literal `none`, never an omission — so a missing field is malformed, mechanically distinct from an empty one. In-session subagents inherit choirmaster's session id as the producing session; the attribution field carries whose draft it was. Refuses when it cannot determine the producing session, and when a field a downstream check requires — review evidence for a gated class — has no source. The trailer-absence audit is a standing job with an owner and a cadence (S16): choirmaster runs it at each handoff ceremony's scrub step, and on demand, scanning main history for commits with missing or malformed trailers; a hit files a `boss-review` issue naming the commits — the mechanical form of the raw-push residual being hit. |
| Declared-vs-actual paths | Refuses when the landing's actual diff touches paths outside the declared set. The declared set = the promotion request's path list PLUS each declared import's destination path; choirmaster's own landings pass an explicit path set to `land` — nothing declares implicitly (boss-ruled 2026-07-21, package-review S14). When current-main reconciliation must touch paths beyond the declared set, the script COMPUTES the extension itself from the actual three-way merge — draft base, draft, current main — and records the computed set in the trailers; a self-declared extension is never accepted, and an actual-diff path beyond declared-plus-computed refuses the landing (S8). |
| Replay protection | Refuses a promotion request whose `request-id` already appears in a landed commit's trailers — one request lands at most once. |
| Entry manifest | Quarry imports run through the throat's import command, which copies the component, computes content digests, and WRITES the `entry-manifest.md` line itself — injected, not remembered. The command assembles the whole component in a temporary tree, verifies destination digests there, and atomically installs the verified result — a crash or a downstream refusal leaves the working tree untouched, and no partial component ever lands (boss-ruled 2026-07-21, package-review S17). Transformations are recorded as executable recipes in the manifest line, so idempotency is mechanical: re-running replays the recipes and converges byte-for-byte, never depending on a human's memory of what was done. The destination digest is computed AFTER all transformations, at landing time, against the exact tree being landed (adopted 2026-07-21 from the prior-art cross-check `https://github.com/nedschorus/nedschorus/issues/5`: Google third-party first-change bundling, Mozilla executable vendoring manifests, Buildroot digests). Line fields: source revision and paths, content digest, destination paths, admission date, purpose, why existing successor code was insufficient, transformations applied. This field list is THE manifest schema — one normative schema, to which README principle 6 and `entry-manifest.md`'s own header defer (boss-ruled 2026-07-21, package-review S6). One component per landing; every import is no-upstream-sync (the quarry is frozen). The check refuses a landing whose DECLARED quarry imports lack their matching manifest line in the same landing. Residual: an UNDECLARED import done by hand outside the import command remains mechanically undetectable (a declared-but-hand-copied import IS refused, for its missing manifest line) — caught by the boss's reading as manifest-less quarry-shaped code. A digest likewise attests the import command's own copy, not independent quarry provenance: a hand-fabricated manifest line carries a digest that verifies nothing against the quarry — same residual class, same catch. |
| Subsystem hygiene | Refuses when a file's path and name carry no token from the known set (the set = top-level subsystem directory names currently in the repo), or when two-plus same-token files sit directly in one top-level root that should hold a subsystem subdirectory (the lazy-subdirectory rule). A token not in the known set is NOT a refusal: the landing proceeds and the script files a `boss-review` issue ("new subsystem, or synonym drift?"). Test-naming rules are enforced at writing time by the writing skills, not here. |
| Gate evidence | Dormant until the boss designates a gated artifact class; the evidence format is defined at designation. Once designated: refuses a landing of that class carrying no review evidence. (Distinct from the deferred PR flow: gating verifies review happened before landing; PR flow would park the artifact in a GitHub review state — that machinery stays out unless the boss wants it.) |
| Current-main validation | For promotions: refuses only what cannot be reconciled. The job applies the draft onto current main; if files changed on main since the draft's base commit conflict with it, the multi-step job reconciles; when reconciliation would require the companion's intent, the landing is refused with the defect and the companion re-drafts against current main. |

## Landing mechanics (boss-ruled 2026-07-21, package-review S14)

- **Clean base.** `land` refuses when the canonical checkout carries dirty paths or local commits outside the declared set — every landing is a known clean base plus exactly the declared changes. Local-only ceremony commits from `handoff.py` are expected ancestors, not dirt: they carry valid trailers and ride the landing's push (the S2 ruling).
- **One landing at a time.** A lockfile in the canonical checkout serializes landings: a second `land` — main thread or promotion subagent — refuses with the lock holder named, never interleaves. Parallel drafting stays free; only the landing moment serializes.
- **Non-fast-forward recovery, defined.** When the push is rejected because origin/main moved: fetch, reconcile the landing onto current main, rebuild the exact candidate tree, rerun ALL checks against it, then push. Never a raw push, never a retry on stale checks.

## States

| State | Meaning | Exit |
|---|---|---|
| EDITING | Uncommitted work in a checkout (either agent's) | Choirmaster: `throat.py land`. Companion: promotion request → PROMOTION-PENDING. |
| PROMOTION-PENDING | A companion draft whose request awaits pickup | Choirmaster handles it at its next turn: lands it, or refuses it with the named defect → back to the companion's EDITING for revision; either outcome is recorded as the mandatory disposition entry keyed by `request-id` (S15). Bound: pickup latency is one choirmaster turn; the request sits in the boss-readable mini-comms log, so a stalled request is visible, never silent. |
| LANDED | On origin/main | — |

(The earlier COMMITTED state collapsed into `land` — the script commits and pushes in one call.)

## Reading discipline

Any checkout other than the canonical one pulls before drafting. The mechanical backstop is current-main validation at landing time.

## Relationship to the quarry's git design

The quarry's `git-clean-slate-plan.md` (read-only reference, `~/Projects/nedlern/docs/working/proposed/`) designed the many-writer version of this problem. nedschorus imports from it only the workflow-rules idea (rewritten for the states above) and the protection-as-lock idea (reduced to one keyholder). The repo's own git config is minimal and stated here, not imported: `user.name` / `user.email` for choirmaster, `useConfigOnly` so no global identity leaks in. Never imported: the three GitHub Apps, the credential helper, per-agent branches, the PR pipeline for ordinary work, the parking states. Deferred with reopening triggers: PR flow for a designated class (trigger: the boss wants a review-parked class); multi-writer machinery (trigger: a genuine third head or true co-writing); deploy handling (trigger: the first long-running process); App-or-CI-identity-as-sole-writer remote enforcement with compare-and-swap landing (the ref updates only if main still equals the validated base SHA) — the rung whose real value is making the CHECKS mechanical, not changing the writer count (trigger: the standing trailer-absence audit firing — see the Provenance check; a main commit lacking a valid provenance trailer is the mechanical form of the residual being hit — or the boss admits the rung on the prior-art evidence in `https://github.com/nedschorus/nedschorus/issues/5`, shortlist item 1).
